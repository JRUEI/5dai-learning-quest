DAY 1 · WHITEPAPER READING · 01 / 18

The New SDLC
with Vibe Coding

這份 Google × Kaggle 課程白皮書不是在說「用 AI 幫忙寫幾行程式」，而是在描述軟體開發如何從手工生產，變成由人設計系統、代理執行工作、持續驗證成果的全新流程。

閱讀目標：理解模型只是能力的一部分；真正的差異在於 Context、Harness、Evals 與人類的判斷。

02 / 18 · WHY NOW

AI 已經跨過「能不能寫」的門檻

  • 基準測試顯示，前沿模型在軟體工程任務的表現大幅進步。
  • 它們能理解程式碼、推理、呼叫工具、修改檔案，並完成較長的任務鏈。
  • 因此，開發問題正從「AI 能否寫 code」轉向「如何讓它在真實專案中可靠工作」。

能力提升不會自動帶來可靠性；工作系統的設計才會。

03 / 18 · AGENT REFRESHER

什麼是 Agent？

  • Agent 是一個持續迴圈：接收任務 → 理解狀態 → 選擇動作 → 使用工具 → 觀察結果 → 再調整。
  • 核心元件包括：模型、指令、工具、Context、記憶與執行環境。
  • 它不是一次性輸出答案的聊天機器人，而是可以在環境中推進工作的系統。

把 Agent 當成「能操作工作台的協作者」，而非搜尋引擎。

04 / 18 · VIBE CODING

Vibe Coding：從描述意圖開始

  • 你先用自然語言說明目標、使用者體驗與限制，再讓 AI 產出可運行的實作。
  • 它特別適合探索、原型、個人工具與速度優先的情境。
  • 但任務一旦牽涉品質、安全、既有架構或長期維護，就必須加入更嚴謹的規格與驗證。

Vibe Coding 是光譜的一端；不是免除工程責任的捷徑。

05 / 18 · THE SPECTRUM

從聊天到代理化工程的光譜

  • Chat / Autocomplete：人主導每一步，AI 提供局部輔助。
  • Vibe Coding：人描述「想要什麼」，AI 快速生成並反覆修正。
  • Agentic Engineering：人提供目標、護欄與驗收標準；Agent 在工具環境中執行較完整的工作流程。

自主性越高，越需要清楚的可驗證規格與安全邊界。

06 / 18 · CONTEXT ENGINEERING

Context 是 Agent 的工作記憶

  • Context 包含目前任務、專案結構、規則、先前決策、工具輸出與使用者回饋。
  • 好的 Context 讓模型知道「現在在哪裡、應遵守什麼、下一步怎麼判斷」。
  • 錯誤、過期或過量的 Context，會讓模型偏離任務，這就是 context rot。

Prompt 只是一句話；Context Engineering 是設計整個資訊供給系統。

07 / 18 · STATIC & DYNAMIC CONTEXT

不是所有資訊都該塞進對話

  • Static context：長期規則，例如架構原則、程式風格、專案指令。
  • Dynamic context：依任務即時取用，例如目前檔案、測試結果、文件片段、資料庫狀態。
  • 白皮書提出六類常用 Context：指令、知識、工具、狀態、回饋與記憶。

只提供「此刻解題真正需要」的資訊，才能保持判斷品質。

08 / 18 · AGENT SKILLS

Skills：漸進揭露的專業知識

  • Skill 是可重複使用的任務說明與資源包，例如部署、測試、設計系統或資料處理。
  • Agent 先看 Skill 的摘要；只有在需要時才讀取完整步驟與參考資料。
  • 這種 progressive disclosure 降低 Context 雜訊，也讓能力能被持續維護。

把知識模組化，讓 Agent 像查工具書一樣按需使用。

09 / 18 · NEW SDLC

傳統 SDLC 正在被重新分配

  • 傳統流程把需求、設計、實作、測試、部署、維護切成多人交接的階段。
  • Agent 能跨越這些階段：讀需求、產出設計、修改程式、執行測試、準備部署。
  • 流程不會消失；它會從「人工交接」轉為「可被代理執行與驗證的迴圈」。

SDLC 的價值不再是排隊流程，而是持續把意圖轉成可驗證結果。

10 / 18 · EARLY PHASES

需求、規劃、設計仍然是人類強項

  • Agent 可以協助彙整訪談、提出 user story、列出假設與產生方案比較。
  • 但「該解哪個問題」、「誰承擔風險」、「什麼算成功」需要人做取捨。
  • 好的需求文件要能讓人與 Agent 都讀懂：目標、限制、邊界、非目標與驗收方式。

越早把意圖說清楚，後面越少用昂貴的返工來修正。

11 / 18 · BUILD & QUALITY

實作速度提升，驗證變成瓶頸

  • Agent 可在既有架構中完成樣板、重構、文件與多檔案修改。
  • 測試、評估（evals）、人工審查與可觀測性，決定產出能否被信任。
  • 每一次失敗都應成為新的測試案例，形成「品質飛輪」。

在 AI 時代，最有價值的不是生成更多，而是更快判斷哪些可用。

12 / 18 · MAINTENANCE

維護與演化：軟體從不真正完成

  • Agent 可協助讀取 issue、追蹤相依套件、偵測弱點與提出修補建議。
  • 但自動變更若沒有 review、回滾與監控，也會放大技術債與安全風險。
  • 生產系統需要把「部署後觀察」納入開發迴圈。

速度讓改動更容易；工程紀律讓改動可逆、可追蹤、可承擔。

13 / 18 · FACTORY MODEL

從工匠模式走向軟體工廠

  • 工匠模式依賴個人記憶與手工操作；產能很難複製。
  • 工廠模式把可靠做法編碼為流程、工具、測試、環境與治理。
  • Agent 可以在這樣的系統中規模化，但前提是系統本身有清楚的標準。

可擴張的不是「某個人很會提示」，而是團隊可重複使用的工作系統。

14 / 18 · HARNESS ENGINEERING

Harness：把模型變成可靠同事的外骨骼

  • Harness 包含 instructions、rules、tools、sandbox、tests、CI、permissions 與 observability。
  • 模型負責推理與生成；Harness 決定它能看見什麼、能做什麼、怎麼被驗收。
  • 白皮書的關鍵觀點：模型只是能力來源的一部分，Harness 決定它在組織中能否安全產生價值。

不要只問「用哪個模型」；也要問「它在什麼系統裡工作」。

15 / 18 · CONDUCTOR

人類角色：Conductor，而不是逐行打字員

  • Conductor 定義目標、拆解工作、分配代理、檢查輸出與處理例外。
  • 它需要理解系統設計、產品權衡、風險、成本與使用者價值。
  • 這不是離開技術，而是把技術能力升級成「編排整個交付系統」。

人類的稀缺價值，從輸入指令轉向設定方向與作最後判斷。

16 / 18 · THE 80% PROBLEM

Agent 擅長 80%，剩下的 20% 更關鍵

  • 例行實作、轉換格式、補文件與初步測試，可能大幅被自動化。
  • 真正困難的是模糊需求、跨系統決策、例外處理、品質定義與責任歸屬。
  • 把工作拆成小、可驗證、可回饋的單位，是讓 Agent 真正有效的基礎。

不要把 80% 當成終點；用它騰出的時間解決真正影響成果的 20%。

17 / 18 · ECONOMICS & RISK

新經濟模型：低 CapEx、高 OpEx

  • 起步成本下降：小團隊能用模型與工具快速做出原型。
  • 營運成本上升：token、推理、評估、監控、人工 review 與安全治理都要持續投入。
  • Context 管理、Skill 重用與模型路由，是控制成本與品質的槓桿。

便宜地生成，並不等於便宜地營運；把品質與成本一起設計。

18 / 18 · STARTING POINT

今天就能開始的三件事

  • 在專案建立明確的 AGENTS.md：記錄規則、驗證方式與不可碰觸的邊界。
  • 先挑低風險、可驗收的任務，要求 Agent 執行測試並解釋結果。
  • 把成功做法整理成可重用的 Skills、模板與自動化流程。

白皮書的結論：未來不是人對 AI，而是具備好 Harness 的人與 AI 團隊，對上沒有系統的開發流程。
